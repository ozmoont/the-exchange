# H1.5 — Virtual Fleet Outbound Flow

*Locked per `STRATEGY.md` decision #12. Implements iCabbi BDD Epic 2.*
*Author: Franko · Status: in progress*

## Problem

Today The Exchange only handles the **inbound** direction: an external originator (FreeNow Dummy, future partners) POSTs a booking, we route to an iCabbi recipient. We've validated that path end-to-end with COID 1102 staging.

The other direction — **outbound**, where iCabbi has no internal driver coverage and wants to send a booking to the network — doesn't work yet. iCabbi has been confirmed to register us as a "virtual fleet" on their Networking Engine (commercial confirmation received). When they hit a no-coverage event they call our fleet API. We need the endpoints they'd call.

Without this, the demo story is one-sided. With it, the Exchange demonstrates value in both directions.

## Out of scope

- **The `/quote` fan-out** with parallel partner-availability queries (BDD Section 1.3, Epic 1.2). Future H2 work — for H1.5 we accept the booking and rank candidates from metadata.
- **`PATCH /bookings/:id` edit-before-allocation** (BDD Section 1.3). Most bookings don't get edited; defer.
- **HMAC signing of iCabbi → Exchange requests.** iCabbi's UI doesn't yet expose signing config (same constraint as inbound webhooks). Use Bearer token auth instead. Migrate when iCabbi adds signing.
- **iCabbi-side fleet registration UI changes.** That's iCabbi's side. We just stand up the endpoints they'll call.
- **The mapping layer.** Per decision #13, hand-coded adapters for MVP. We translate iCabbi's payload to our `NormalisedBooking` directly.

## Acceptance criteria

1. **Inbound endpoint `POST /api/icabbi/bookings` accepts a valid offer** with Bearer token auth, creates a transit row with the calling iCabbi tenant as the originator, hands the booking to `receiveBooking()` for routing, and returns an iCabbi-shaped acknowledgement within 2 seconds (per BDD NFR).
2. **Bearer token authentication identifies the originator partner** — each iCabbi tenant has its own token stored in encrypted credentials. Tokens are surfaced on the integration page so an admin can give them to iCabbi.
3. **`POST /api/icabbi/cancellations` accepts a cancellation** for a previously-offered booking, cascades to the recipient adapter's `cancelBooking`, and marks the transit cancelled.
4. **Status updates flow back to iCabbi** — when the recipient advances the booking lifecycle, we POST a status update to the iCabbi tenant's documented fleet-status endpoint. Best-effort; failures land in `outbound_webhook_deliveries` for retry.
5. **No code changes affect the existing inbound flow.** All H1.5 work is additive — new endpoints, new helpers, no schema changes to existing tables.
6. **Loop detection** — a booking offered by iCabbi tenant X must NEVER be re-routed back to iCabbi tenant X. Enforced in `rankCandidates()` with an existing-known check on `originatorPartnerId`.

## Endpoint contracts

### `POST /api/icabbi/bookings`

**Headers:**
- `Authorization: Bearer <token>` — identifies the originator partner
- `Content-Type: application/json`

**Body (best-guess shape; subject to confirmation by iCabbi):**
```json
{
  "booking_id": "iCabbi-side-booking-id",
  "trip_id": "<optional iCabbi trip id>",
  "booking_type": "ASAP" | "PREBOOK",
  "scheduled_at": "<ISO 8601, required if PREBOOK>",
  "pickup": {
    "lat": 51.507,
    "lng": -0.128,
    "address": "Trafalgar Square, London"
  },
  "dropoff": {
    "lat": 51.470,
    "lng": -0.454,
    "address": "Heathrow Terminal 5"
  },
  "passenger": {
    "name": "Jane Doe",
    "phone": "+447700900000"
  },
  "passenger_count": 1,
  "vehicle_type": "saloon",
  "fare_estimate": 42.50,
  "currency": "GBP",
  "instructions": "Ring buzzer 3",
  "notes": "Account customer, VIP"
}
```

**Response 200:**
```json
{
  "status": "accepted",
  "exchange_transit_id": "uuid",
  "estimated_eta_minutes": 8,
  "recipient_partner": "FreeNow"
}
```

**Response 400:** malformed body / missing required fields
**Response 401:** Bearer token missing or invalid
**Response 422:** routing failed (no coverage in network)
**Response 409:** duplicate `booking_id` from same originator (idempotency)

### `POST /api/icabbi/cancellations`

**Headers:** same auth as above.

**Body:**
```json
{
  "booking_id": "iCabbi-side-booking-id",
  "reason": "passenger_cancelled" | "driver_no_show" | "other",
  "reason_detail": "<optional freeform>"
}
```

**Response 200:** cancellation propagated to recipient (best-effort)
**Response 404:** booking_id not found for this originator

### Bearer token issuance

When an admin clicks **Connect** on `/partners/[id]/integration` for an iCabbi tenant, we also generate a Bearer token (in addition to the webhook signing secret). Stored in encrypted `credentials.inboundBearerToken`. Surfaced **once** in the green banner alongside the webhook signing secret — admin copies it and gives it to iCabbi for their fleet-config Authorization header.

Token format: 48-byte base64url string. Rotation supported via existing rotate-secret action (rename to "Rotate webhook + bearer secrets" or split into two actions — TBD).

## Loop detection

`rankCandidates()` already filters by `bilateral_allow` rules. We add one rule: **the originator partner is automatically excluded from being a recipient for their own bookings.** Already implicit via the bilateral rule check (no rule from X to X), but make it explicit + comment so future engineers don't accidentally remove it.

For cross-tenant loop detection (iCabbi tenant A sends to us, we try to route back to iCabbi tenant B which then has no coverage and tries to send back to us): if a transit's `originatorPartnerId` is an iCabbi-kind partner, exclude all iCabbi-kind partners from candidate set. Forces real external partner fulfilment for iCabbi-initiated bookings. Avoids the "hot potato" failure mode where two iCabbi tenants bounce a booking back and forth.

## Status sync back

When the recipient advances the lifecycle (`accepted`, `driver_assigned`, `en_route`, `on_board`, `completed`), the existing `forwardStatusUpdate()` flow runs. We extend it: if the originator is an iCabbi-kind partner, we POST the status update to iCabbi's documented fleet-status endpoint on their tenant.

iCabbi item #5 in `ICABBI_DEPENDENCIES.md` is still partially open — we don't yet know the exact path iCabbi expects status events on for fleet-mode (vs. the partner-mode webhook URL we use today). For H1.5 ship a best-guess POST to `${apiBaseUrl}/fleet/status` with the same Bearer auth in reverse (we sign with iCabbi's Secret-Key, send App-Key in header). If that fails, the existing `outbound_webhook_deliveries` records the failure with the status code and message — observable in `/webhooks`.

## Verification

When iCabbi (or a mocked caller using their staging Bearer token) POSTs a valid booking to `/api/icabbi/bookings`:

- 200 response within 2s with `exchange_transit_id`
- Transit row created with `originator_partner_id = COID 1102` (or whichever iCabbi tenant)
- Routing engine picks FreeNow Dummy (or another non-iCabbi partner) as recipient
- `/transits/<id>` shows the routing trace
- Loop detection check confirmed: no iCabbi partner appears in the routing candidates
- When recipient advances status, an outbound delivery to iCabbi's fleet-status URL appears in `/webhooks` inspector

## Risks / open questions

1. **iCabbi's exact inbound payload shape isn't documented.** I'm building against a best guess. First real call from them will tell us what to adjust — same iterative approach we used for webhook events.
2. **iCabbi's fleet-status endpoint path isn't confirmed.** Best-guess `/fleet/status` may need a tweak when we see the first 404/401.
3. **Bearer token revocation** — current model rotates by re-clicking Connect. Doesn't handle "kick this token NOW" scenario. Defer to H4 (production hardening) unless a security need surfaces.
4. **Rate limiting on `/api/icabbi/*` routes** — reuse the existing Postgres-backed rate limiter with a per-partner key like `inbound_icabbi:${partnerId}`. Same 60/min default.
5. **What happens if iCabbi calls us with a booking_id we've seen before** (retry, network blip)? Same idempotency pattern as the webhook ingest — return 409 + the original transit id, no second transit created.
